# Credit Risk Scoring API

An explainable credit default risk scorer built as a fintech portfolio project. Submit a credit card applicant's financial profile and get back a default probability, the top factors driving that score (via SHAP), and a plain-English explanation generated by Llama 3.1.

**Stack:** FastAPI · XGBoost · SHAP · Groq/Llama 3.1 · asyncpg/PostgreSQL · React + Vite · Recharts

---

## Setup

### Prerequisites
- Python 3.11+ and pip
- Node.js 18+
- PostgreSQL 14+ running locally
- A [Groq API key](https://console.groq.com) (free tier is sufficient)

### 1. Clone and install

```bash
git clone <repo-url>
cd "credit-risk-scoring-api"

# Python backend
python -m venv .venv && source .venv/bin/activate
pip install -r backend/requirements.txt

# macOS: XGBoost needs OpenMP
brew install libomp

# React frontend
cd frontend && npm install && cd ..
```

### 2. Configure environment

```bash
cp .env.example .env
# Edit .env — set DATABASE_URL and GROQ_API_KEY
```

### 3. Set up the database

```bash
createdb credit_risk
psql credit_risk -f backend/db/schema.sql
```

### 4. Get the dataset and train the model

The model file (`backend/ml/model.ubj`) is committed to the repo — **you can skip this step** and go straight to step 5.

If you want to retrain from scratch:

```bash
python scripts/download_data.py       # downloads ~5 MB XLS from UCI
python -m backend.ml.train            # ~30 seconds, saves model.ubj
```

### 5. Run

```bash
# Terminal 1 — backend
source .venv/bin/activate
uvicorn backend.api.app:app --port 8000

# Terminal 2 — frontend
cd frontend && npm run dev
```

Open **http://localhost:5173** (or 5174 if 5173 is in use).

## API Reference

### `POST /predict`

```json
{
  "credit_limit": 50000,
  "sex": 2,
  "education": 2,
  "marriage": 2,
  "age": 30,
  "pay_0": -1,
  "pay_2": -1,
  "pay_3": -1,
  "pay_4": -1,
  "pay_5": -1,
  "pay_6": -1,
  "bill_amt1": 5000,
  ...
  "pay_amt6": 1000
}
```

Response includes `risk_score` (0–1), `risk_label` (Low/Medium/High), `top_factors` (SHAP), and `llm_explanation`.

### `GET /predictions/{id}`

Returns a stored prediction with its full feature set.

### `GET /model/feature-importance`

Returns global feature importance (mean |SHAP|) ranked across 500 test-set samples, suitable for rendering a Recharts bar chart.

## Model Card

See [`backend/ml/MODEL_CARD.md`](backend/ml/MODEL_CARD.md) for full details. Summary:

| | |
|---|---|
| **Algorithm** | XGBoost gradient-boosted trees |
| **Dataset** | UCI Default of Credit Card Clients (Taiwan, 2005) — 30,000 rows |
| **Target** | Default payment next month (binary) |
| **AUC-ROC** | ~0.78 |
| **Average Precision** | ~0.55 |
| **Class imbalance** | 22% positive rate; handled via `scale_pos_weight=3.5` |
| **Explainability** | SHAP TreeExplainer (exact, not approximate) |

**Why AUC-ROC and not accuracy?** The dataset has 78% non-defaulters. A model predicting "never default" scores 78% accuracy while being completely useless. AUC-ROC measures discrimination at every threshold; Average Precision additionally penalises false positives on the majority class.

## Limitations & Ethical Considerations

This is a portfolio demonstration. It must not be used for any real credit decision.

**Data limitations**
- Taiwan credit card holders, 2005. Not representative of any current lending population.
- No bureau data, employment type, loan purpose, or collateral — variables central to real underwriting.
- Static dataset: no concept drift detection, no retraining pipeline.

**Model limitations**
- AUC-ROC of 0.78 is reasonable for this feature set but well below what production systems achieve with richer data.
- Threshold at 0.5 is arbitrary. A real deployment would tune the threshold to a specific precision/recall operating point based on the cost of false negatives vs. false positives.
- No calibration check — the raw probability output may be systematically over- or under-confident.

**Fairness**
- Features include `sex`, `education`, `marriage`, and `age` — some of which are protected attributes in many jurisdictions.
- No disparate impact analysis has been performed. A production system would require fairness auditing across demographic groups before deployment.
- Feedback loops: a biased model creates biased outcomes, which skew future training data.

**Explainability scope**
- SHAP values explain *the model's reasoning*, not objective creditworthiness. If the model has learned spurious correlations, SHAP will faithfully explain those too.
- The LLM layer narrates SHAP output; it does not add independent judgement. Its constraints (temperature=0.2, factor-only prompt) are designed to prevent hallucination of reasons not in the data.

**Legal**
- Lenders in many jurisdictions (EU, US ECOA/FCRA) are legally required to provide adverse-action notices with specific reasons. The SHAP factors shown here are a technical starting point, not a compliant adverse-action notice.
